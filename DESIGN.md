# Design

## Overview

This repository publishes a research-backed prompt engineering catalog. The
product source of truth is `README.md`; the generated web site is a static,
browser-friendly rendering of that source plus a small allowlist of root
Markdown pages.

The site build has three goals:

- Preserve README content and anchors for readers who use GitHub.
- Render a safe static site for Vercel.
- Generate crawl and AI discovery artifacts from the same source.

## Content Source Of Truth

`README.md` remains the catalog contract. The static site must not introduce
catalog text, recipe counts, evidence tiers, or source claims that are not
present in the README or generated by the checked scripts.

Only explicitly allowlisted root Markdown files may become public pages. The
renderer currently allows `README.md`, `CONTRIBUTING.md`, `SECURITY.md`, and
`CODE_OF_CONDUCT.md`. Repo-internal stewardship files, including `.agents/` and
`AGENTS.md`, remain source files rather than site routes.

## Render Pipeline

The web build reads allowlisted Markdown pages, renders them with `markdown-it`,
adds GitHub-style heading anchors, sanitizes HTML, rewrites links for the static
site, copies CSS assets, and writes output under ignored `public/`.

Pagefind indexes `public/` after the HTML build. The canonical build command is
`pnpm run build`, which cleans old output, builds the HTML, runs Pagefind, and
checks generated artifacts.

`public/` is a single-writer build artifact in a shared workspace. Commands that
clean, build, index, or check generated output must be serialized; parallel
agents may run source-only checks at the same time, but only one lane should
mutate `public/`.

## Trust And Sanitization

README content, fetched pages, generated files, and tool output are treated as
untrusted evidence. Markdown rendering must pass through the sanitizer before
publication. Allowed HTML and style attributes are intentionally narrow and
should only expand when a README feature has a concrete rendering need.

Generated HTML checks must reject obvious unsafe snippets (including bare
executable `<script>` blocks after allowlisting JSON-LD and the analytics config
JSON script), raw GitHub alert markers, unpublished relative Markdown links, and
relative `.agents/` links. Raw Markdown discovery files such as `llms-full.txt`
may preserve README source syntax, but browser HTML must be clean.

### Analytics and Content-Security-Policy

Production `vercel.json` ships a **strict** CSP (`script-src` / `connect-src`
`'self'`) while analytics defaults to `WEB_ANALYTICS_PROVIDER=none`. That pairing
is intentional. Enabling PostHog or Umami without widening CSP will break third-
party scripts in browsers. Before enabling analytics:

1. Keep the provider at `none` until CSP is updated.
2. For PostHog, allow `https://*.posthog.com` in `script-src` and `connect-src`
   (see [PostHog CSP docs](https://posthog.com/docs/advanced/content-security-policy)).
3. For Umami, allow the self-hosted script and API hosts.
4. Optional build gate: set `WEB_ANALYTICS_CSP_ENFORCE=1` so `web/build.mjs`
   loads `vercel.json` CSP and runs `assertAnalyticsCspCompatible` against the
   configured provider before writing `public/`. Unset (default) skips the check.
   Unit coverage lives in `web/test/analytics.test.mjs`.

Do not pre-open third-party hosts in CSP while analytics remains disabled.

### Recipe vs pattern ItemList split

Home JSON-LD uses dual `ItemList` graphs (48 prompt recipes + 43 pattern notes).
Classification currently treats h4 headings with an icon `<img>` as recipes and
h4s without as pattern notes. Generated-site checks assert both counts. Avoid
adding decorative images to pattern-note headings without updating the extractor.

## Link Policy

Public routes are generated only for allowlisted root Markdown sources. Links to
those pages are rewritten to static-site routes. Links to unpublished repo-local
Markdown files are rewritten to GitHub `blob/main` source URLs so readers can
inspect stewardship docs without the site publishing those files as pages.

External `http`, `https`, and `mailto` links are preserved. Hash-only links are
preserved. Non-Markdown relative assets are rewritten by route depth for nested
pages.

## GitHub Alerts

The README uses GitHub alert blockquote syntax such as `> [!TIP]`. The generated
site renders those blocks with `markdown-alert` classes and visible alert labels,
without exposing the raw `[!TIP]` marker text in HTML.

Supported alert kinds are `TIP`, `NOTE`, `IMPORTANT`, `WARNING`, and `CAUTION`.

## Vercel Deployment

Vercel is the deployment target. `vercel.json` configures the project as a
static site with `framework: null`, `pnpm run build`, and `public` as the output
directory. `trailingSlash: true` keeps served URLs aligned with the generated
canonical URLs and sitemap entries. GitHub Actions remains a quality workflow
and does not deploy with Vercel tokens.

Canonical URLs are resolved from `WEB_BASE_URL` first, then Vercel system
variables, then the local preview URL. The local fallback is intentionally not a
production domain; do not hardcode an unverified live URL into
`web/site.config.mjs`. For production, set `WEB_BASE_URL` to the verified
deployment origin or expose Vercel system URL variables before building.

## Validation Matrix

`AGENTS.md` is the canonical validation source. Local command shortcuts in
`justfile`, package scripts, pre-commit hooks, and GitHub Actions must wrap the
same checks rather than define competing success criteria.

Use `just validate-fast` for deterministic local checks and `just validate` for
full parity, including browser smoke, external link checks, badge URL checks,
and pre-push hooks.

## Non-Goals

- No dynamic server or serverless functions.
- No GitHub Pages deployment workflow.
- No Vercel CLI deploy requirement in local validation.
- No publication of `.agents/` or arbitrary nested Markdown as site pages.
- No hand-edited generated README badge counts or ShieldCN URLs.
